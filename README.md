# Reaction Roulette

A local-first meme/reaction GIF search and clipboard tool. Paste a URL, search by natural language later, click to copy the URL back into Discord.

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│  apps/windows_app/        — Flutter desktop UI (Forui)          │
│  ├─ lib/main.dart         — loading screen → Coordinator init   │
│  ├─ lib/inference_factory.dart — creates ONNX embedders         │
│  ├─ lib/inference_impl.dart — CLIP BPE tokenizer + ONNX wrappers│
│  ├─ lib/state/signals.dart — signals bridging Coordinator ↔ UI  │
│  └─ lib/widgets/          — sidebar, search bar, grid, scaffold │
├─────────────────────────────────────────────────────────────────┤
│  packages/meme_collector_core/  — pure Dart backend             │
│  ├─ lib/models.dart       — Reaction entity + JSON codec        │
│  ├─ lib/storage.dart      — metadata.json + atomic rename (FFI) │
│  ├─ lib/inference.dart    — abstract TextEmbedder/ImageEmbedder │
│  ├─ lib/search.dart       — SIMD vector index + RRF fusion      │
│  ├─ lib/ingest.dart       — URL parser + ffmpeg + pipeline      │

```

## How it works

### Startup flow

1. `main()` calls `runApp()` immediately → loading screen appears
2. `_initCoordinator()` runs in the background:
   - Creates `Storage` (ensures directories exist)
   - Loads `metadata.json` (source of truth for all reactions)
   - Creates `ClipTextEmbedder` (CLIP text tower ONNX, ~127 MB) for query embedding
   - Detects ffmpeg on PATH
   - Spawns **search isolate** (owns `VectorIndex` + `KeywordIndex`, pure Dart)
   - Tells search isolate to load vectors from disk into memory
   - Creates `IngestPipeline` (runs in main isolate — ONNX can't cross isolate boundaries)
3. `isInitialized` signal flips to `true` → UI swaps to `HomeScaffold`

### Data model

```
storage/
├── metadata.json                    — source of truth (URLs, tags, usage)
├── downloads/{id}.{ext}             — original downloaded files
├── thumbnails_static/{id}.webp      — static WebP previews (ffmpeg)
├── thumbnails_animated/{id}.webp    — animated previews (optional)
├── embeddings_text/{producer}__{version}/{id}.f32.bin
├── embeddings_image/{producer}__{version}/{id}.f32.bin
└── ocr/{id}.txt                     — OCR text (when OCR is implemented)
```

- **metadata.json** is small (~5 MB for 10k entries), debounced writes (500ms), atomic rename via `MoveFileExW` FFI on Windows
- **Derivative files** are named by reaction UUID, model version in directory name (easy re-embedding when models upgrade)
- **No SQLite/Drift/Isar** — JSON + files only, rebuildable from disk

### Search pipeline

When the user types a query:

1. `MySearchBar` debounces 150ms → calls `performSearch(query)`
2. `performSearch()` calls `coordinator.search(query)`:
   - Embeds query with CLIP text tower (~70ms) → 512-dim Float32List
   - Sends `SearchRequest` to search isolate
3. Search isolate runs parallel retrieval:
   - **Text vector ANN** — SIMD dot product against all text embeddings (~2ms for 10k vectors)
   - **Image vector ANN** — cross-modal: CLIP text query vs CLIP image embeddings
   - **Keyword search** — inverted index over title + tags + ocrText
4. **RRF fusion**: `score(d) = Σ 1/(60 + rank_i(d))` for each retriever
5. **Usage boost**: `+ 0.05 * log(1 + usageCount)`
6. Returns top 50 `SearchResult`s → `searchResults` signal → grid rebuilds

### Ingest pipeline

When a reaction is added (currently via `coordinator.addReaction(url)`):

1. **Create entry** — UUID v7, normalize URL (Tenor/Giphy/Discord/direct), dedup check
2. **Immediate text embedding** — embed title+tags with CLIP → write to disk → send to search isolate → reaction is instantly searchable
3. **Download** — `dio.download()` to `.tmp` → rename on success (crash-safe)
4. **Probe** — `ffprobe` → mimeType, dimensions, isAnimated
5. **Thumbnail** — `ffmpeg` → static WebP (256px wide)
6. **Text embedding** — re-embed with any new metadata
7. **Image embedding** — CLIP vision tower on the thumbnail (~100ms) → write to disk → send to search isolate
8. **OCR** (when implemented) — PP-OCRv5 det + rec → enrich text embedding

Each stage emits `IngestProgressEvent` → `ingestProgress` signal → grid tile shows progress bar.

### Click behavior

- **Left click** — copies reaction URL to clipboard + increments usage count + shows "Link copied" toast (Discord-like behavior)
- **Right click** — TODO: context menu (copy URL / file path / file bytes / open in browser / pin / delete)

## Setup

### Prerequisites

- Flutter 3.44+ (beta channel, matches SDK constraint `^3.13.0-103.1.beta`)
- Rust toolchain (only needed if re-adding model2vec — currently removed)
- ffmpeg + ffprobe on PATH (for thumbnails + probing)
- Windows 10 1607+ (for atomic file rename via MoveFileExW)

### Install

```bash
git clone https://github.com/Foxyz1302/meme_collector_project.git
cd meme_collector_project
flutter pub get

# Download model files (~478 MB)
dart run scripts/download_models.dart

# Run the app
cd apps/windows_app
flutter run -d windows
```

### Model files

Downloaded to `assets/models/` (gitignored):

| File | Size | Purpose |
|---|---|---|
| `clip_text_fp16.onnx` | 127 MB | CLIP text tower (query + text embedding) |
| `clip_vision_fp16.onnx` | 176 MB | CLIP vision tower (image embedding) |
| `clip_tokenizer.json` | ~1 MB | CLIP BPE tokenizer spec |
| `ppocr_det.onnx` | 84 MB | PP-OCRv5 text detection (not yet wired) |
| `ppocr_rec.onnx` | ~8 MB | PP-OCRv5 text recognition (not yet wired) |
| `ppocr_dict.txt` | ~1 KB | Character dictionary for OCR |

**Note:** model2vec was removed — its Native Assets build hook is incompatible with Flutter 3.13 beta SDK. CLIP text tower is used for all text embeddings instead (~70ms vs ~1ms, still within 150ms debounce).

## Coordinator API (how the UI talks to the backend)

```dart
// Search
coordinator.search("sarcastic cat")        // → Future<List<SearchResult>>
coordinator.hotlist(limit: 60)             // → Future<List<SearchResult>>

// Add/delete
coordinator.addReaction(url: "...", title: "...", tags: [...])
coordinator.deleteReaction(id)
coordinator.pinReaction(id)                 // force download for dead URLs

// Usage
coordinator.incrementUsage(id)             // bump usage count + lastUsedAt
coordinator.updateMetadata(id, title: ..., tags: ...)

// Live updates
coordinator.progressStream                  // Stream<IngestProgressMsg>
coordinator.completeStream                  // Stream<IngestCompleteMsg>
coordinator.failedStream                    // Stream<IngestFailedMsg>

// Path resolution (for Image.file)
coordinator.resolvePath(relativePath)       // → absolute path
coordinator.storagePath                     // → storage root

// Lifecycle
coordinator.flushNow()                      // force-write metadata (on app exit)
coordinator.dispose()                       // shut down isolates
```

## Key design decisions

1. **No database** — metadata.json + files on disk. JSON is ~350ms to parse for 10k entries, trivially rebuildable, inspectable in any text editor. No migrations ever.

2. **URL is primary identity** — Discord-like behavior: click copies the URL, file is just a preview derivative. Pinning model for dead-URL resilience.

3. **Core is pure Dart** — no Flutter deps. The app provides concrete ONNX implementations via `InferenceFactory`. Future server can import the same core with different embedder implementations.

4. **Search isolate only** — ONNX sessions can't cross isolate boundaries (native pointers). Search isolate uses pure-Dart `VectorIndex` (safe). Ingest pipeline runs in main isolate (I/O-bound, brief ONNX bursts).

5. **Model versioning in directory names** — `embeddings_text/model2vec__potion-base-32M/{id}.f32.bin`. When a model upgrades, re-embed into a new directory, update config, old directory stays as fallback. No destructive migration.

6. **FP16 over INT8** — INT8 uses ConvInteger ops that CPU EP doesn't support. FP16 works everywhere, half the size of FP32, ~2× GPU throughput when we eventually get GPU working.
